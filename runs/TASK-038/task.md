## Description
Add a read-only Quest overview endpoint for Mobile/Frontend:

`GET /api/v1/quest/overview?user_id=...`

The endpoint should compose the existing Missions daily/monthly/streak data into a screen-friendly response. Weekly, Event, and Invite tabs should be present as ordered placeholders with `active=false` and empty `items` so Mobile can build the tab UI without waiting for later engines.

VIP/Game level header data must come from `Games-Labs-User` via gRPC `GetProfile`. The current `GetProfileResponse.Data` contract exposes `level` and `exp`, not `vip_level`; derive `vip_label` as `VIP{level}` in Missions for phase 1. If the User client fails, degrade gracefully with `user.partial=true` and still return the Quest overview.

## Acceptance Criteria
- [ ] `shared-lib/proto/missionspb/missions.proto` has an additive `GetQuestOverview` RPC with grpc-gateway annotation `GET /api/v1/quest/overview`.
- [ ] `missionspb` generated artifacts are regenerated and not manually edited.
- [ ] `Games-Labs-Missions` exposes HTTP `GET /api/v1/quest/overview?user_id=...` and gRPC dispatch for `GetQuestOverview`.
- [ ] Response `tabs` is an ordered array in this exact order: `daily`, `weekly`, `monthly`, `event`, `invite`.
- [ ] Daily and Monthly tabs are populated from existing Missions progress/streak/monthly data where available.

- [ ] User header uses User gRPC `GetProfile.Data.level` and derives `vip_label = "VIP{level}"`.
- [ ] If User gRPC/profile lookup fails, endpoint still returns 200 with `user.partial=true`.
- [ ] No `GetVipTiers` RPC or `/api/v1/vip/tiers` route is added to Missions.
- [ ] Focused service/handler tests pass, including user-client degradation.
